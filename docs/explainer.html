The problem behind the models

These aren't toy models on toy data. They were trained on real operational data from a live inside-sales call system — every call the sales team places or receives, logged with who dialed, when, and whether it connected.

The business problem // scale breaks triage

The team makes hundreds of calls a day, and only ~41% connect. Once you're dialing at that volume, placing calls isn't the bottleneck — knowing when and by whom to dial is. Rep time is the scarce resource; spending it on numbers that won't pick up, at hours nobody answers, is pure waste.

What we set out to predict

  • Connect likelihood (model 1). At the moment of dialing — before the call is placed — estimate the probability it will be answered, from dial-time features (rep, line, hour, day). That score ranks when and by whom to call, so effort flows to the calls most likely to connect.
  • Operational volume (model 2). Forecast call throughput over time — how many calls to expect next — so staffing and capacity can be planned instead of guessed.

Both are genuine supervised-learning problems with honest difficulty: a balanced but thin-signal label, and a small sample — which is exactly why the honesty section matters here and isn't decoration.

Why TensorFlow / Keras // one engine, two problem shapes

The two problems span two different data shapes — structured columns and a time series. TensorFlow/Keras was the deliberate choice:

ReasonWhat it buys

in-graph preprocessingNormalization and StringLookup live inside the model, so the saved artifact ingests a raw feature dict. Kills train/serve skew — a production concern, not a notebook nicety.

The one-liner

A high-volume call operation produces more dials than a human can optimize. These models turn its history into a ranking signal (which calls will connect) and a capacity forecast (how much volume is coming) — built on TensorFlow because one framework, with production-grade in-graph preprocessing, covers every data shape the problem throws at it.

§1

The one flip that makes ML click

You already know how to write a function: you supply the rules, the caller supplies the input, out comes the output. Machine learning inverts the middle step. You can't write the rules — they're too messy (how do you score a sales call's odds of connecting?) — so instead you hand the machine a pile of example input→output pairs, and an algorithm derives the rules for you.

A "model" is therefore just a function whose body you didn't write. Training is the process that writes that body. Inference is calling the finished function like any other.

§2

Anatomy of the model — the function itself

Before we can train anything, we need the parameterized function whose "rules" are just numbers we can tune. Here is every part, bottom-up.

Tensor // the only data type

A tensor is an n-dimensional array of numbers — that's it. A single number is a 0-D tensor (scalar), a list is 1-D (vector), a spreadsheet is 2-D (matrix), a batch of images is 4-D. Everything flowing through the model — inputs, weights, intermediate results, outputs — is a tensor. The "Tensor" in TensorFlow is literally this: the library is a machine for pushing tensors through math, fast, on a GPU.

Weights & biases // the tunable knobs

These are the numbers the training process is allowed to change — the "learned" part of the model. A small model has thousands; a large language model has hundreds of billions. Untrained, they're random, so the function outputs noise. Learning = finding good values for these knobs. Nothing else in the model changes.

The neuron & the activation function // where nonlinearity comes from

A neuron computes a weighted sum of its inputs plus a bias (w·x + b — a dot product, pure linear algebra), then passes it through a fixed activation function f. That function is the crucial twist: without it, stacking layers would just collapse into one big linear equation (a line), no matter how many you stack. The nonlinearity — e.g. ReLU (max(0,x), "pass positives, zero out negatives") or sigmoid (squash any number into 0–1) — is what lets the network bend and represent complex, curved decision boundaries.

Programmer's model

A layer is one line: output = activation(W @ input + b) — a matrix multiply, an add, a squish. A "deep" network is just function composition: L3(L2(L1(x))). Depth buys expressiveness the same way composing small functions buys you complex behavior.

Layer, network, forward pass

A layer (Keras Dense) is a vector of neurons sharing the same inputs. A network is layers chained so each one's output feeds the next. Running data through, input→output, is the forward pass — that's inference, and it's also step one of every training iteration.

§3

The training loop — how the knobs get tuned

This is the heart of it, and it's an optimization loop you could have written yourself. Start with random knobs; repeatedly measure how wrong the model is and nudge every knob in the direction that reduces the wrongness.

Loss function // the number you minimize

The loss collapses "how wrong were all the predictions" into a single number. Lower = better. The whole loop exists to drive this down. Choice of loss depends on the task: binary cross-entropy for yes/no probabilities (our two classifiers), mean squared error for predicting a real number (our forecaster). It's your objective function — the thing an optimizer optimizes.

Gradient // which way is downhill

The gradient is the vector of partial derivatives of the loss with respect to every knob — it answers, for each knob, "if I increase this a hair, does the loss go up or down, and how steeply?" It points in the direction of steepest increase, so we step the opposite way.

Backpropagation // computing the gradient efficiently

A network is a deep composition of functions, so its derivative is computed by the chain rule — run backward from the loss through each layer, multiplying local derivatives as you go. That's all "backprop" is: the chain rule, applied right-to-left, reusing intermediate results so it's fast. You never do this by hand — automatic differentiation in TensorFlow records every operation of the forward pass and replays it in reverse.

Programmer's model

You're on a foggy hillside wanting the lowest point. You can't see the valley, but you can feel the slope under your feet (the gradient) and step downhill. Repeat thousands of times. That's gradient descent; backprop is just the efficient way to feel the slope on a very high-dimensional hill.

Optimizer, learning rate, epoch, batch

TermWhat it is
optimizerThe rule for turning gradients into knob updates. Adam (what we used) keeps per-knob momentum & adaptive step sizes — robust default.
learning rateGlobal step-size multiplier. The single most important dial. We used 1e-3.
batchWe don't compute the loss on all data at once — we use small chunks (e.g. 16 rows) per step. Faster, and the noise actually helps generalization.

§4

Our two models — same recipe, two data shapes

Both are the identical loop above. The only thing that differs is the shape of the data going in and the layer types that suit it.

Model 1 — Tabular classifier // yes/no from columns

The connect-rate model predicts whether a call will be answered, from structured columns known at dial time (which rep, inbound/outbound, which line, hour-of-day, day-of-week). A network eats numbers, so numeric columns get normalized (rescaled to mean 0, so no single big-numbered feature dominates) and each category — the rep, the line — gets one-hot encoded (a 0/1 column per value, because "rep A" isn't numerically greater than "rep B"). Both steps are Keras layers, so the saved model ingests a raw feature dict — no external scaler to reproduce. That feeds two Dense layers → one probability. We run it head-to-head against gradient-boosted trees, the honest baseline for tabular data.

Model 2 — LSTM forecaster // predict the next number in a sequence

An LSTM (Long Short-Term Memory) is a layer with an internal memory cell and "gates" that decide what to remember, forget, and output as it reads a sequence step by step. That memory is what lets it learn trend and seasonality. We fed it via a sliding window — a trick to manufacture supervised examples from one long series — then forecast recursively by feeding each prediction back in as the next input.

§5

Doing it honestly — the part that's pure engineering discipline

Every "ML gotcha" below has an exact analogue in ordinary engineering. This section is what separates a real model from a misleading demo — and it's your strongest interview material, because you can reason about it.

Overfitting // memorizing instead of generalizing

An early version scored 0.97 on training data but far lower on unseen data — it had memorized the training rows like a lookup table rather than learning a rule, so it failed on anything new. This is the identical bug to hardcoding answers to your exact test cases: the suite passes, the code is broken.

Train / validation / test split // never grade on data you trained on

We hold out data the model never sees in training and score only on that — an integration test on fresh inputs. Validation = the slice used to decide when to stop; test = the final, untouched exam.

Data leakage // accidentally seeing the answer key

We fit the number-scaler and category-encoders on the training portion only, and for the time series we never let a future month touch the past. Compute those stats over the whole dataset and information about the test set "leaks" into training, inflating your scores dishonestly — like benchmarking with the answer sheet open. And a feature you won't have at prediction time (a call's duration, before it's placed) can't be a feature at all.

Class imbalance // the "always return false" trap

When one outcome is rare — say only 30% of examples are positive — a model that always predicts the majority scores 70% accuracy and is completely useless. The fix is class weights — make errors on the rare class cost more — so the optimizer can't ignore the minority. It's also why we don't report raw accuracy.

The metrics — and why not "accuracy"

ROC‑AUC
ranking quality
Pick a random answered & missed call: how often does the model rank the answered one higher? 0.5 = coin flip, 1.0 = perfect. Ours ≈ 0.60.
AP
avg precision
Precision-recall summarized; judged against the 0.41 base rate, so beating 0.41 means real signal.
RMSE
forecast error
Average error magnitude in real units. Only meaningful vs a baseline — we beat naive by 19%.
The honest summary

Our connect-rate numbers are modest because the dial-time signal is genuinely thin and the sample small — time-of-day and which rep dials carry most of it. We reported that via cross-validation and baseline comparisons instead of cherry-picking one split. Understanding why a model is mediocre is worth more than a pretty number you can't explain.

§6

The engine & the API — TensorFlow vs Keras

Two layers, one install. TensorFlow is the engine: it represents your model as a computation graph, runs the tensor math fast on CPU/GPU, and — the killer feature — does automatic differentiation so the training loop's gradients come for free. Keras is the high-level API you actually write in: Layers, a Model, and compile / fit / predict. You describe what the model is; TF works out how to run and differentiate it.

Two setup facts worth knowing

  • Python 3.11 in a virtualenv. TensorFlow doesn't publish builds for the system's Python 3.14 yet — pure dependency-version pinning, exactly like locking a runtime.
  • .keras saved files. The trained knobs serialized to disk, so you load and call the function later without retraining — a compiled artifact.
One line for the interview

"The model is a parameterized function fitted to data by gradient descent. TensorFlow handles the graph, GPU execution, and autodiff; Keras is the API I describe the model in. I was disciplined about train/test separation, leakage, class imbalance, overfitting, and cross-validated baselines — so the numbers are honest, and I can explain exactly why they are what they are."
